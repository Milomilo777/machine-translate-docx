1. root cause:
- `sys.exit(2)` was mistakenly executed unconditionally outside of the Exception handling block that is used when `docx.Document(word_file_to_translate)` fails parsing the file, thereby crashing the entire python CLI command regardless of the document's validity.
- `--source-language` was mistakenly mandatory, even though the internal translation engine relies on `--srclang`.
- `--action` argument was defaulting to translate, but other AI actions needed normalization.

2. files changed:
- `src/machine-translate-docx.py`

3. exact patch summary:
- `PATCH 1`: Replaced `--source-language`'s strict `required=True` logic into `required=False`, matching standard options. Allowed `--action` to accept `double`.
- `PATCH 2`: Ensured argument normalization executes properly. Normalized `args.action` explicitly and fallback `args.srclang` to English ('en') if omitted or mapped from `--source-language`.
- `PATCH 3`: Shifted the `sys.exit(2)` block from the unconditional global scope logic immediately beneath the exception handler back *into* the exception block correctly to prevent it from preemptively crashing legitimate word file translations. Also bypassed a breaking `get_monitors()` CLI startup error when run headlessly in containers with a try/except.

4. why sys.exit(2) happened:
- `sys.exit(2)` was completely outside of the indentation of `except:` handler where docx parsing failures were gracefully handled, which resulted in unconditional aborts whenever execution reached that block.

5. why this is the minimum safe fix:
- We simply modified `src/machine-translate-docx.py` with an exact indentation adjustment for `sys.exit(2)` inside the exception block. Argparse default resolution was normalized explicitly and safely post-execution. No architectural changes were made to dispatch logic since processaiaction() already handles 'double' effectively, maintaining strict compatibility with the user CLI interface context.
