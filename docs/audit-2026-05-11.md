## Phase 1 — Raw findings (one bullet per item)

Severity tags: **CRIT** (must-fix), **HIGH**, **MED**, **LOW**.

### Reliability — backend (`local_launcher.py` + `src/`)

- **R-1 (HIGH)** — `LocalState.job_procs` grows unbounded. The dict is
  written when a subprocess starts and only pruned by the 1-h job
  cleanup thread; nothing removes the entry on `proc.wait()` return.
  *local_launcher.py: ~1084 + ~1134.*
- **R-2 (HIGH)** — `proc.stdout` reader loop in `_run_real_backend` does
  not explicitly close the pipe on exception. If the parser raises
  mid-loop, the kernel buffer may not be drained and the child blocks.
  *local_launcher.py: ~1095-1132.*
- **R-3 (MED)** — `_archive_failed_job`'s outer try/except swallows any
  `mkdir`/permission error and prints once; the operator sees a single
  warning and the failure folder is silently absent for that job. No
  retry, no fallback path. *local_launcher.py: ~1217-1275.*
- **R-4 (MED)** — Bare exception in `_append_subscriber()` returns the
  raw exception text into the JSON response, leaking internal error
  state to the user. *local_launcher.py: ~161-162.*
- **R-5 (MED)** — `_sanitize_filename()` does not cap length. Win32
  silently truncates / fails on > 255 chars. *local_launcher.py:
  ~104-109.*
- **R-6 (LOW)** — `cancel_job` reads job status under lock then kills
  outside the lock; concurrent cancel + worker exit could race.
  *local_launcher.py: ~378-402.*
- **R-7 (LOW)** — Selenium engine error paths in `engines/google.py`
  swallow with `except Exception: pass` and return empty string —
  caller has no way to distinguish "element missing" from "network
  timeout" or "browser crash".
- **R-8 (LOW)** — Python `print()` of `deepl_account_email` reaches
  launcher stdout (and the failure-archive `stdout.log`). The
  `--silent` flag mitigates in production but not in dev. *src/engines/
  deepl.py: ~248.*

### Frontend — UX + correctness

- **F-1 (CRIT)** — Legacy `index.ejs` still lists `gpt-5.4` and
  `gpt-5.5-mini` in the AI-model dropdown. Both will now hit B-004's
  `VALID_AI_MODELS` rejection at upload time, but the form gives no
  pre-flight hint. *index.ejs: ~80-85.*
- **F-2 (HIGH)** — Structured `[FAIL] reason=<token>` is parsed by the
  launcher into `jobs[id].error`, but v2's `pollStatus` simply throws
  `data.error || 'Translation failed'`, losing the token. The user
  sees a generic message. *web/v2/app.js: ~339-343.*
- **F-3 (HIGH)** — Select-arrow background gradient in styles.css uses
  hardcoded 45° / 135°; in `[dir="rtl"]` the position flips but the
  arrow direction does not. Persian users see a visually wrong
  chevron. *web/v2/styles.css: ~430-440.*
- **F-4 (MED)** — Progress bar gradient is hardcoded `linear-gradient(
  90deg, ...)`; in RTL the bar still grows toward the left. Cosmetic.
  *web/v2/styles.css: ~534-539.*
- **F-5 (MED)** — Theme toggle in legacy `/` is not persisted; v2
  stores in `localStorage('v2.theme')`. Switching pages resets the
  theme. *index.ejs: ~479-506.*
- **F-6 (MED)** — `loadAndRenderContent()` writes via `textContent` on
  every field (safe), but the `<li>` template for files in
  `renderFileList()` uses `innerHTML` for the wrapper + `textContent`
  for the text — mixed style. *web/v2/app.js: ~218-235.*
- **F-7 (LOW)** — No offline banner. If `navigator.onLine === false`
  the user clicks "Translate" and only learns from a network error.
- **F-8 (LOW)** — Form `<label>` wraps the `<select>` implicitly
  rather than using `for=` — works for most ATs but not standard.

### OpenAI / cost surface

- **C-1 (HIGH)** — Polisher's pricing table is missing `gpt-5.5`. The
  default model's polish cost is reported as `$0.00` in the sidecar.
  *src/openai_tools/polisher.py: ~324.*
- **C-2 (HIGH)** — Splitter has no `cached_tokens` extraction and no
  cached-tier prices, so its $-figure on a cache hit is overstated.
  *src/openai_tools/splitting.py: ~73-115.*
- **C-3 (LOW)** — `line_count_reconciler.py` makes API calls without
  any cost capture. Volume is small; effectively invisible in the
  sidecar.
- **C-4 (DESIGN, ✓)** — Server-side cache key is
  `sha256(payload + lang + engine + model)` and intentionally **does
  not** include `splitTranslate` / `splitEngine` so a re-run with a
  different splitter can reuse the translation. Confirmed by the
  `_materialise_cached_output()` call site.
- **C-5 (✓)** — `call_with_retry` correctly differentiates HTTP 429
  (retried with exponential backoff) from `BadRequestError` (no
  retry). 3 attempts, 1 / 2 / 4 s.

### Future low-cost user features (cheap wins)

- **U-1 (HIGH-VALUE / LOW-COST)** — Cancel-running-job button. Backend
  already exposes `POST /cancel/<id>`; v2 does not wire it.
- **U-2 (HIGH-VALUE / LOW-COST)** — Pre-run cost estimate. The
  pricing tables already live in `translator.py` / `polisher.py`; a
  new `GET /pricing` endpoint exposes them; the frontend multiplies
  `file.size / 4` (rough char-to-token) by the rate. Two-line backend.
- **U-3 (MED-VALUE)** — Download `_log.json` sidecar from the v2 page
  after a chatgpt-polish run. Backend already serves it under
  `/download/<name>` — pure frontend wiring.
- **U-4 (MED-VALUE)** — Per-session cumulative cost in the v2 footer.
  Read the sidecar JSON after each run, accumulate in localStorage.
- **U-5 (LOW)** — `navigator.onLine` banner.
- **U-6 (LOW)** — Theme persistence on legacy `/` for parity with v2.

### Hygiene + docs + tooling

- **H-1 (HIGH)** — `.claude/launch.json` is currently produced by the
  Claude Preview tool and lives inside `.claude/` which is excluded by
  `.gitignore` — but it's worktree-specific personal config and should
  stay excluded. The current tracked-file scan confirms only
  `.claude/rules/*` is tracked, nothing else. ✓ no leak today; the
  rule is fragile (a future commit could accidentally `git add` a
  worktree config). Strengthen: explicit pattern in `.gitignore`.
- **H-2 (MED)** — Dead commented imports in
  `src/machine_translate_docx.py` (~14 + ~20 lines, CJK / Yandex /
  Thai / parsivar) — inert but noisy.
- **H-3 (MED)** — `todo.txt` at repo root is from 2022 and references
  retired features. Stale.
- **H-4 (MED)** — `_sync_globals_from_ctx`'s docstring already
  describes its whitelist nature (closed in W-1) but does not list
  every mirrored field — a checklist would help future edits.
- **H-5 (LOW)** — Three `requirements*.txt` files with overlapping
  pins, no README explaining which to use.
- **H-6 (LOW)** — README.md at repo root is one line; CLAUDE.md is
  the de-facto README.
- **H-7 (LOW)** — Several `_real_test/` files were tracked into git
  by an earlier flow; current `.gitignore` now covers it but some
  artefacts may still be in history (harmless — outputs only).
- **H-8 (LOW)** — Heavy `print()` usage with semantic prefixes
  `[INFO]` / `[DIAG]` / `[FAIL]` is fine; one place still emits raw
  account email (R-8).

### Test coverage
